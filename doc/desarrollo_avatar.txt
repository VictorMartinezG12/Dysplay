OBJETIVO GENERAL


Implementar el módulo Avatar para DysPlay.

DysPlay es una plataforma web educativa para niños con dislexia.

El Avatar NO es solamente una personalización visual.

El Avatar será:

- Compañero educativo.
- Asistente global del sistema.
- Mensajero de eventos.
- Representación visual del usuario.
- Elemento persistente durante toda la navegación.

Debe estar disponible globalmente desde base.html.


REQUISITO PRINCIPAL


El Avatar NO debe calcular lógica de negocio.

NO debe:

- Calcular recompensas.
- Calcular experiencia.
- Calcular progreso.
- Decidir cambios de nivel.
- Modificar monedas directamente.
- Gestionar correos.
- Gestionar autenticación.

Su función es únicamente:

- Mostrar.
- Reaccionar.
- Comunicar.


ARQUITECTURA


Implementar una arquitectura desacoplada.

NO permitir:

Modulo -> Avatar

Permitir:

Modulo -> Evento -> Avatar

Diseñar un sistema basado en eventos.

Ejemplos:

PRONUNCIACION_CORRECTA
PRONUNCIACION_INCORRECTA
OBJETO_RECONOCIDO
OBJETO_NO_RECONOCIDO
HISTORIA_COMPLETADA
NIVEL_COMPLETADO
LOGRO_DESBLOQUEADO
ERROR_CONEXION
ERROR_CARGA

El Avatar debe escuchar eventos y reaccionar.

Los módulos NO deben conocer la implementación interna del Avatar.

INTEGRACION CON CAMARA INTELIGENTE


La cámara inteligente reconoce objetos.

El Avatar NO debe reconocer objetos.

Debe reaccionar a eventos emitidos por la cámara.

Ejemplo:

OBJETO_RECONOCIDO

Avatar:
- animación feliz
- mensaje contextual

OBJETO_NO_RECONOCIDO

Avatar:
- animación pensativa
- mensaje de ayuda


INTEGRACION CON RECOMPENSAS


El Avatar NO calcula recompensas.

Debe aceptar payloads externos.

Ejemplo:

{
  "tipo_evento":"RECOMPENSA_GANADA",
  "monto":50,
  "unidad":"monedas",
  "mensaje":"¡Ganaste 50 monedas!"
}

El Avatar únicamente:

- muestra mensaje
- reproduce animación
- celebra
